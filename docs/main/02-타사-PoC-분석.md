## 2. 잘 된 점 (가져갈 것)

| 항목 | 내용 |
|------|------|
| **정보 구조** | 실시간 관제 / 확인 대기 / CCTV / 통계 / 히스토리 / 설정 — **역할 분리는 합리적** |
| **이벤트 3연동** | 알림 리스트 ↔ 3D 구역 ↔ CCTV — **스펙이 요구한 플로우의 뼈대는 있음** |
| **AI 오버레이** | CCTV 위 노란 구역 폴리곤 + 인원/혼잡 % — **프로젝트 NVR 연동 방향과 맞음** |
| **안전선·혼잡·낙상** | API·필터·통계 탭까지 **기능 범위는 넓음** |
| **확인 워크플로** | 확인 대기 → 모니터링 화면에서 보기 → 확인 완료 — **운영 프로세스 고려** |

→ **기능 체크리스트 관점에서는 PoC 요건을 많이 채웠다.** 발주 측 기대와의 간극은 “기능 없음”보다 **체감 품질·관제 UX·디지털 트윈 느낌** 쪽일 가능성이 큼.

## 3. 화면 기준 보완·개선 여지 (추정)

> 주의: 아래는 **추정**이다. 발주 측 피드백이 오면 우선순위를 조정할 것.

### 3.1 화면에서 바로 확인되는 점

| # | 보완 여지 | 근거 (스크린샷) |
|---|----------|----------------|
| A | **3D 표현이 프로토타입 수준에 가까움** | 흰색 박스 매스 + 위성지도 / 또는 **정사 파노라마 사진**에 빨간 면 붙인 수준 |
| B | **한 화면에 정보가 많음** | 좌·중·우 동시 풀가동 + 3D 위 플로팅 라벨(357%, 391%) + PIP + 범례 + 뷰 버튼 |
| C | **혼잡도 % 해석이 어려움** | 361%, 391% — UI에 **원시값(Raw) 그대로** 노출, 운영자 혼란 (→ §4.4-1) |
| D | **3D와 데이터 연결이 약함** | 알림은 리스트, CCTV는 우측 고정 3칸 — **한 눈에 “어디서 무슨 일”인지** 파악이 어려움 |
| E | **관제실 톤·무드** | 일반 SaaS 대시보드 + 네온 그린 — 철도 본사 관제실에 기대되는 **차분함**과 거리 있음 |
| F | **철도 관제 요소가 적음** | 열차·전광판·승강장 안전선 **3D 연출 없음** — KPI 숫자만 상단에 존재 |

### 3.2 운영·프로젝트 관점에서 고려할 점

| # | 보완 여지 | 설명 |
|---|----------|------|
| G | **CCTV 그리드 = NVR 복제** | CCTV 탭이 공간 맥락 없는 2D 그리드 — “트윈 안에서 보자”는 목표와 어긋날 수 있음 |
| H | **통계·히스토리가 3D와 단절** | BI 차트·리스트만 — **어디서** 많이 터졌는지 공간 인지가 어려움 |
| I | **설정 UX 확장성** | XYZ·스트림 ID 수동 입력 — 카메라 50~80대 확장 시 부담이 클 수 있음 |
| J | **성능·완성도** | 스크린샷만으로는 불명 — 데모 시 끊김·로딩이 있었다면 체감에 영향 |

### 3.3 스펙과의 정합

견적/PoC 문서에 이미 적혀 있음:

> *기능 검증 중심 → **전체 화면·UX 재검토·재기획** 필요*

→ 발주 측도 **기능 나열 PoC**가 아니라 **관제 제품 수준**을 기대했을 가능성.

## 4. 그래피카 PoC 구현 방향

### 4.1 방향 요약 (참고)

```text
타사 접근: 기능 중심 2D 관제 + 3D(또는 사진) 보조
그래피카 제안: 3D 중심의 역사 관제 화면
```

### 4.2 화면별 그래피카 제안

| 타사 화면 | 그래피카 (P0~P1) |
|----------|------------------|
| 좌측 알림 항상 풀너비 | **플로팅 글래스 패널** — 3D **full-bleed 고정**, 패널만 오버레이 (Push 금지) |
| 우측 CCTV 3~4칸 고정 | **컨텍스트 1채널** + 인근 2~3 썸네일 |
| 혼잡 300%+ 표시 | **Adapter 정규화 레이어** → 0~100% 또는 4단계 (Raw는 UI에 안 뿌림) |
| 흰 박스 + 위성지도 | **역사 매스** (한 톤 다크) — 위성지도는 P2 또는 제거 |
| 파노라마 + 빨간 면 | **Three.js 바닥 히트맵** + 구역 mesh |
| 안전선 = KPI 숫자만 | **플랫폼 가장자리 라인 mesh** + 침범 시 색·알람·fly-to |
| 열차 없음 | **TAGO 전광판 + 3D 열차 진입·정차** (철도 관제 요소) |
| API 상태등 나열 | PoC는 **성능 위젯(FPS/draw call)** 으로 기술 신뢰 |
| 설정에서 XYZ 입력 | PoC는 **`devices.json`** — 프로젝트 **3D Drag & Drop 노코드 배치** 로드맵 (§4.4-3) |

### 4.3 데모 시나리오 (S1~S3, 약 3분)

| 순서 | 화면 | 동작 |
|------|------|------|
| 1 | 전체 뷰 | 역사 매스 + 이벤트 3D 마커 |
| 2 | 평시 (S1) | TAGO 전광판 + 3D 열차 접근 연출 |
| 3 | 안전선 침범 (S2) | 알람 클릭 → fly-to → 안전선 강조 → CCTV 1채널 |
| 4 | 화장실 위급 (S3) | 동일 패턴 — fly-to + 재실 패널 |

### 4.4 구현 보완 (외부 리뷰 반영)

분석은 맞지만, **UI 설계**에 아래를 반드시 녹인다. (`03-UX` §2.3, §5.5)

#### 4.4-1 혼잡도 391% — Raw 값 해석

타사가 391%를 쓴 이유는 **UI 실수만**이 아닐 가능성이 크다.

- 현장 AI 카메라·혼잡 솔루션이 **설계 기준 인원 대비 현재 인원 비율(Raw)** 을 그대로 보냄  
- 타사는 그걸 **가공 레이어 없이** 화면에 뿌림 → 관제사 혼란

**그래피카 제안 (구현: `adapters/normalize.ts`, NF-09):**

```text
현장 Raw (300%+)  →  Adapter 정규화  →  UI (0~100% 또는 여유/주의/심각)
```

→ 현장 Raw 스펙을 이해하고 **Adapter에서 정규화**하는 구조 (NF-09).

#### 4.4-2 접이식 패널 — Push가 아니라 Floating Overlay

패널을 열 때 **3D가 줄어들거나 레이아웃이 밀리면(Push)** 관제사 시각 피로가 큼.  
대형 모니터를 멀리서 보는 업무 특성상 **화면이 덜컹거리는 것**을 극도로 싫어함.

| Push (금지) | Floating Overlay |
|--------|---------------------|
| 패널 열림 → 3D viewport 폭/스케일 축소 | **3D canvas 크기·위치 고정** (항상 full-bleed) |
| 레이아웃 전체가 재배치 | 반투명 **글래스 패널**만 좌·우·상단에 **겹침** |
| | `pointer-events` 분리 — 3D Orbit 유지 |

**비기능 가치:** “장시간 구동 시 시각적 피로 최소화” (NF-10 보완)

#### 4.4-3 설정 탭 대안

타사 설정 탭(XYZ·go2rtc 수동 입력)의 **의도**는 “운영자가 개발자 없이 카메라 위치 수정”.

PoC에서 설정 **화면은 만들지 않는다.** 좌표는 `devices.json`으로 관리하고, 프로젝트에서는 **3D Drag & Drop 노코드 편집** (P2: `F-22`)으로 확장한다.
